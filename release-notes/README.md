# Release notes

Per-version Markdown that the release workflow renders into:

1. **Sparkle "Check for Updates…" dialog** — the in-app updater fetches an
   HTML rendering of the corresponding `.md` from
   `Opt1-Releases/release-notes/<version>.html`. This is what existing users
   on the previous version see *before* they choose to install the update.
2. **GitHub Release body** on `JAHealey1/Opt1-Releases/releases/tag/v<version>` —
   GitHub renders Markdown natively, so the `.md` is used verbatim.
3. (Indirectly) the Homebrew cask consumes the same release as a download
   target — no notes there.

## Convention

- One file per release: `release-notes/<MAJOR>.<MINOR>.<PATCH>.md` (matching
  `MARKETING_VERSION` and the git tag).
- The first line should be a level-1 heading (`# Opt1 X.Y.Z`).
- Use lists, code fences, and links freely — both Sparkle (after rendering)
  and GitHub support standard CommonMark + GFM.
- Keep it short and user-facing. This is what someone sees when they're
  deciding whether to update; lean on imperative bullet points like
  "Fixed crash on Sonoma when overlay is offscreen" rather than commit
  history.

## Workflow

1. Before tagging, write `release-notes/<version>.md` and commit it on the
   same PR (or commit) as the version bump in `Opt1.xcodeproj`.
2. Push the tag (`git tag vX.Y.Z && git push origin vX.Y.Z`).
3. The release workflow renders the Markdown to HTML, ships the HTML in the
   `Opt1-Releases` repo for Sparkle, and uses the raw Markdown as the
   GitHub Release body.

If `release-notes/<version>.md` does not exist when a tag is pushed, the
release workflow **fails**. That is intentional — every release should have
deliberate notes attached, not an autogenerated placeholder.
